设计文档的设计

System-context-diagram​

在许多文档中，系统上下文图可能非常有用。这样的图表将系统显示为更大的技术景观的一部分，并允许读者在他们已经熟悉的环境中将新设计置于上下文中。

APIs​

如果正在设计的系统公开了一个 API，那么勾勒出该 API 通常是一个好主意。然而，在大多数情况下，人们应该承受将正式界面或数据定义复制粘贴到文档中的诱惑，因为它们通常很冗长，包含不必要的细节并且很快就会过时。相反，专注于与设计相关的部分及其权衡。

Data storage​

Degree of constraint​

这样的文档可能范围很广，但它也需要快速定义一组规则，允许放大一组可管理的解决方案。 另一方面，系统对可能的解决方案进行了很好的定义，但如何将它们组合起来以实现目标并不明显。

Alternatives considered​

Cross-cutting concerns​

在这里，您的组织可以确保始终考虑某些横切问题，例如安全性、隐私和可观测性。这些通常是相对较短的部分，用于解释设计如何影响关注点以及如何解决关注点。团队应该标准化他们案例中的这些问题。

由于其重要性，Google项目需要有专门的隐私设计文档，并且有专门的隐私和安全审查。虽然审查只需要在项目启动时完成，但最佳做法是尽早与隐私和安全团队合作，以确保设计从头开始考虑它们。对于这些主题的专用文档，中央设计文档当然可以引用它们，而不是详细介绍。
