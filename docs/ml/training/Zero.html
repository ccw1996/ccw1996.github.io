Zero-DP

模型状态在训练过程中通常会消耗大量的显存，原始的DP具有良好的计算/通信效率，但显存效率较差。更具体地说，DP在所有进程中均需要存放整个模型副本，从而导致了冗余的显存消耗。这些方法都静态地维护了整个训练过程中所需的所有模型状态，但并不是所有模型状态在训练过程中都需要。而ZeRO-DP它既实现了DP的计算/通信效率，又一定程度上优化了显存消耗。如图1所示， ZeRO-DP有三个主要的优化阶段，分别对应于优化器状态、梯度和参数的分区。

  1. 优化器状态分片（Pos）：每一个计算设备仅保存对应的分片；
  2. 模型梯度分片（Pos+g）：，每一个计算设备仅保存对应的优化器状态分片与梯度分片；
  3. 模型参数分片（Pos+g+p）：每一个计算设备仅保存对应的优化器状态分片、梯度分片以及参数分片；

ZeRO-DP行为分析​

ZeRO Stage1: 单个计算设备会保存完整的模型参数和梯度，保存1/n的优化器状态分片，单个step详细的过程如下：

  • 前向计算：每一个计算设备拿到对应的数据样本进行前向计算，计算loss；
  • 反向计算：通过前向计算的loss和模型参数本身，计算出模型参数对应的grad，这里每个计算设备均有属于其本身的完整的梯度；
  • 通信组内通信：此处通信为reduce-scatter,即在通信组内所有计算设备上的梯度发送到不同计算设备，每一个计算设备的梯度其中属于对应优化器状态分片的为所有计算设备的平均梯度，注意这里reduce-scatter之后，每一个计算设备有两类梯度：属于对应优化器分片的通信组内平均过的梯度与不属于对应优化器分片的未通信前的梯度；
  • 优化器计算：优化器根据对应分片已在通信组内平均过的梯度去更新对应部分的模型参数，得到更新过的参数，这时候每个计算设备上有1/n的参数是更新后的参数，n-1/n的参数为未更新的参数；
  • 通信组内通信:此处通信类型为allgather，将分布在各个计算设备上更新后的1/n的参数分发到所有计算设备上，即此时所有计算设备上都有了完整的更新后参数；

ZeRO Stage2: 单个计算设备上会保存完整的模型参数，保存1/n的模型梯度与优化器状态分片，单个step详细的过程如下：

  • 前向计算：每一个计算设备拿到对应的数据样本进行前向计算，计算loss；
  • 反向计算：这里相对stage1 有一些不好理解，因为这里每个显卡没办法模型的全部梯度。但所幸，梯度是逐层计算的，当前向计算loss完成后，计算最后一层的梯度，而最后一层的梯度会按某些规则将通信组内所有计算设备中该层的平均梯度发送至某个计算设备，同时删掉通信组内其他计算设备上的该层梯度。这样逐层按上述逻辑，每一层的平均梯度都能属于到通信组内仅一个计算设备，达到每个计算设备仅保存1/n的模型梯度。整体来看，这里可以等同于梯度的整个reduce-scatter的操作（会建立多次链接）；
  • 优化器计算：优化器根据对应分片已在通信组内平均过的梯度去更新对应部分的模型参数，得到更新过的参数，这时候每个计算设备上有1/n的参数是更新后的参数，n-1/n的参数为未更新的参数；
  • 通信组内通信:此处通信类型为allgather，将分布在各个计算设备上更新后的1/n的参数分发到所有计算设备上，即此时所有计算设备上都有了完整的更新后参数；

ZeRO Stage3单个计算设备上保存1/n的模型参数、模型梯度与优化器状态分片，单个step详细的过程如下：

  • 通信组通信+前向计算：此处等同于allgather，即通信组内每个计算设备拿到组内属于其他的参数，进行前向计算，可以按流水线，逐层获取，即第一层前向计算时，保存该层参数的通信设备，将该层参数广播至其他未保存的n-1个计算设备，依次类推，即可完成前向计算，计算出loss；
  • 通信组通信+后向计算+通信组通信：思考下，后向计算之前也要进行等同前向中的模型参数的广播，即每一层归属的计算设备将模型参数广播至其他计算设备，即可计算出对应的梯度，接着需要计算通信组内平均梯度，即需要组内记性依次reduce-scatter即可每个计算设备保存对应的平均梯度与模型参数；
  • 优化器计算：根据计算出的对应分片的平均梯度和对应的模型参数，即可计算出更新后的模型参数；

Zero-R

降低activation显存

  1. activation checkpoint，将很多关键activation消耗显存较多的layer在反向计算时冲计算，activation checkpoint(recompute)是一种常见方法，可以在牺牲33%的重复计算开销的同时，将activation显存减少到其平方根左右。即GPT-2模型的activation显存消耗降至约8GB。但是即使使用activation checkpoint，较大的模型的activation显存仍可能变得非常大。例如，1000亿参数的GPT模型，在批次大小为32时，即使activation checkpoint，也需要大约60GB的显存，activation checkpoint有很多变种地方式结合对模型activation与计算耗时的tradeoff做了很多工作，这里面有相当多的技巧，如果以后有机会可以单独成文讨论；
  2. 当Zero与tensor并行组合使用时，尤其是再比如在GPT中QKV拼接之后的linear计算，tensor并行会将其分在多个计算设备，这里本身activation极大。zero采取需要时通过新增一次allgather来重新计算activation，从而大大减少这里需要的activation显存；

Zero-Offload

ZeRO-Offload通过将数据和计算卸载到CPU来实现大模型训练。为了保持计算效率，它旨在最小化GPU与CPU之间的数据移动，并减少CPU计算时间，同时最大程度地节省GPU内存。相比于使用PyTorch单独训练具有14亿参数模型时达到的30 TFlops/GPU，ZeRO-Offload可以在单个NVIDIA V100 GPU上为具有100亿参数模型提供40 TFlops/GPU。

  1. 圆形节点表示模型状态（parameter16、gradient16、parameter32、momentum32、variance32）；
  2. 矩形节点表示计算（forward、backward、param update）;
  3. 图中的边表示节点之间的数据流，边的权重是在任何给定的训练迭代期间通过它流动的总数据量（以字节为单位）;

为了确定最佳的offload策略，ZeRO-Offload 使用第一原理分析来高效地将这个图切分在 CPU 和 GPU 设备之间。ZeRO-Offload策略优化主要包括三个关键因素：

  1. 需要保证cpu计算负载小于GPU，从而防止CPU成为性能瓶颈：
    1. CPU计算能力比GPU计算能力慢几个数量级，将模型计算图卸载到CPU上会严重限制训练效率。因而，我们必须避免将计算密集部分offload到CPU上；
    2. 分析可知每次模型训练的计算复杂度通常为O（MB），其中M是模型大小，B是有效批处理大小。为了避免CPU的计算成为瓶颈，只有那些计算复杂度低于O（MB）的计算应该被卸载到CPU上，而前向传播和反向传播都具有O（MB）的计算复杂度，适合在GPU上完成，而其余计算，如范数计算、权重更新等，其复杂度为O（M），可以offload到CPU上。基于这个简单的观察，我们将数据流图中的前向节点和反向节点融合成一个单一的超级节点（FWD-BWD），并分配至GPU上；
  2. 保证最小化CPU和GPU存储之间的通信量：
    1. CPU内存带宽至少比CPU和GPU之间的PCI-E带宽快一个数量级，而GPU内存则比CPU内存甚至更快一个数量级。因此，我们必须最小化CPU和GPU内存之间的通信量，以防止PCI-E带宽成为训练性能瓶颈；
    2. 任何模型状态offload策略的最小通信量为4M， 在前向和反向过程合并为单个超级节点的前提下，数据流图中的每个节点都是循环的一部分。因此，对该图的任何分割都需要割掉至少两条边，每条边的边权至少为2M，从而导致总通信量至少为4M（理论最小通信量）；
    3. 任何不将fp32模型状态与其生产者和消费者节点放置与一个计算设备的的切分策略都无法实现最小通信量为4M。这样的切分必须割掉至少一条边，其权重为4M，另一条边的权重至少为2M，导致通信量至少为6M。因此，fp32模型状态（momentum32、variance32和parameter32）必须与Param Update和float2half计算一起放置。这个约束使我们可以将数据流图中的所有上述fp32数据和对应的计算节点视为Update超级节点（无法内部切分）。我们在图2中展示了这个简化的数据流图，其中只包含四个节点：FWD-BWD超级节点、p16数据节点、g16数据节点和Update超级节点；
    4. p16必须与FWD-BWD超级节点放置一起，因为这两个节点之间的边权为4M。分离这两个节点将会增加通信量至6M（即4M + 2M）。由于我们已经将节点FWD-BWD Super分配给GPU以限制在CPU上的计算，因此p16也必须分配给GPU；
  3. 在尽可能保证最小化通信量的前提下，获得更多的显存节省：
    1. 在保证最小化CPU和GPU存储之间的通信量之后， 只剩下g16和Update超级节点需要分配。因此最大化考虑GPU上的显存节省。通过将g16和Update 超级节点都offload到CPU上，最大的内存节省可以达到8倍；通过上述三条原则，最终的计算图切分方案如下：
